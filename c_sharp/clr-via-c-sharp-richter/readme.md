## Выделение памяти и сборка мусора

1. При выполнении оператора new вычисляется размер объекта (учитываются размер всех его полей, указатель на объект-тип и индекс блока синхронизации)

2. Если в поколении 0 управляемой кучи достаточно места для объекта, то переходим к шагу 2. Иначе переходим к сборке мусора на шаге 4

3. По адресу NextObjPtr обнуляется полученное количество байты, вызывается конструктор, указатель NextObjPtr сдвигается. Выделение памяти завершается без сборки мусора

4. Сборка мусора. Все потоки в процессе приостанавливаются. Определяются поколения, в которых должна выполяться сборка мусора

5. Поколения 1 и 2 будут участвовать в сборке мусора только если общий размер объектов в каждом их них уже привысил выбранные лимиты. Чаще всего сборка происходит только в поколении 0, реже в поколения 0 и 1, реже всего во всех поколениях: 0, 1, 2. Это сделано для оптимизации

6. Маркировка (marking phase). Всем объектам проставляется 0 в поле индекса блока синхронизации.

7. Перебираются все корни и объекты, на которые они ссылаются (построение дерева "живых" объектов). Им проставляется 1 в индекс блока синхронизации. У недостижимых объектов остается 0 (алгоритм отслеживания ссылок, а не подсчет ссылок). Тут есть еще одна оптимизация, описанная далее.

8. Если выполняется сборка мусора в младшем поколении, то во время обхода дерева объектов частично игнорируются объекты из старшего поколения. Если объект из младшего поколения ссылается на объект из старшего поколения, то в объекте из старшено поколения проверяется флаг изменения ссылочных полей объекта с момента последней сборки мусора. Если изменений нет, то старший объект и все его ссылочные поля можно игнорировать. Если изменения есть, то в построение дерева включаются объекты по изменившимся ссылочным полям.

9. Сжатие (compacting phase). Все "немусорные" объекты (1 в ИБС) перемещаются к началу кучи, чтобы они занимали в ней смежные блоки памяти. А ссылки на них и NextObjPtr корректируются соответственно

10. Объекты, пережившие сборку мусора в 0 поколении, переходят в поколение 1.

11. Если в результате сборки мусора не удалось освободить достаточно места для нового объекта, то получаем OutOfMemoryException

## Режимы сборки мусора

- Режим рабочей станции - сборщик старается не занимать все ресурсы, оставляет другим приложениям

- Режим сервера - использует все ресурсы, работает с кучей в несколько параллельных потоков

Подрежимы: параллельный (фоновый) / непараллельный (блокирующий). При параллельном подрежиме создается дополнительный поток, выполняющий маркировку в фоне. Сборка мусора в поколениях 0 и 1 выполняется как обычно. Но перед сборкой в поколении 2 увеличивает поколение 0 для размещения текущего объекта, и в фоне начинает выполнятся маркировка. Затем принимется решение выполнять или нет сжатие. Приложение потребляет больше памяти, но увеличивается быстродействие

Для режим рабочей станции по умолчанию применяется параллельный подрежим, для сервера - непараеллельный. Все это можно настраивать (изменять)
